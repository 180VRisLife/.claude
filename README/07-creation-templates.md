# Creation Templates

Copy-pastable prompts for extending the Library system with new templates or domains.

## New Template Type Prompt

Use this to add a new type of code template across all relevant domains (e.g., analytics integration, crash reporter).

```
Add a new template type for [TEMPLATE_NAME] that [TEMPLATE_DESCRIPTION e.g., "provides crash reporting integration for production error tracking"].

Steps:
1. Research best practices for this template type across different languages/platforms
2. Create the baseline template in library/default/file-templates/[template-name].template
3. For each relevant domain, create a domain-specific version:
   - library/ios/file-templates/[TemplateName].swift.template
   - library/macos/file-templates/[TemplateName].swift.template
   - library/visionos/file-templates/[TemplateName].swift.template
   - library/webdev/file-templates/[templateName].ts.template (or .tsx for React components)
   - library/streamdeck/file-templates/[templateName].ts.template
4. Follow existing naming conventions per domain:
   - Swift: PascalCase (e.g., CrashReporter.swift.template)
   - TypeScript: camelCase (e.g., crashReporter.ts.template)
   - Python: snake_case (e.g., crash_reporter.py.template)
5. Each template should:
   - Include header comment: "Auto-generated by /0-workspace - customize as needed"
   - Use placeholders like {{PROJECT_NAME}} where appropriate
   - Follow the domain's existing patterns (check other templates in the same folder)

Skip domains where this template type doesn't apply (e.g., don't create React components for Swift domains).
```

## New Domain Prompt

Use this to add support for a new development platform or framework.

```
Create a new development domain for [DOMAIN_NAME] that [DOMAIN_DESCRIPTION e.g., "supports building Android applications with Kotlin"].

Steps:
1. Create the domain folder structure:
   ~/.claude/library/[domain]/
   ├── file-templates/
   └── settings.local.template.json

2. Research the platform's conventions for:
   - Debug/development logging
   - Git information display
   - .gitignore patterns
   - Environment configuration (if applicable)

3. Create file-templates/ with platform-appropriate templates:
   - DebugLogger.[ext].template - Centralized logging that adapts to DEBUG/RELEASE modes
   - GitInfo.[ext].template - Extracts branch, commit hash, dirty status
   - .gitignore.template - Platform-specific ignore patterns
   - (Optional) Additional templates as needed for the platform

4. Create settings.local.template.json by copying from library/default/ and adjusting if needed

5. (Optional) Create scripts/ folder for platform-specific utilities

6. Update ~/.claude/scripts/init-workspace.py:
   a. Add domain indicators (strong and medium) following existing patterns
   b. Add detection logic in the appropriate priority order
   c. Ensure config file checks are domain-specific

File naming conventions by language:
- Swift/Kotlin: PascalCase (DebugLogger.swift, DebugLogger.kt)
- TypeScript/JavaScript: camelCase (debugLogger.ts)
- Python: snake_case (debug_logger.py)

Validate by running `/0-workspace` in a project of the new domain type.
```

**After creating a domain:** Test it by running `/0-workspace` in a project of that type.
